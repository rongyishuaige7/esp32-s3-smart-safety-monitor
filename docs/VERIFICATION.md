# 验证说明

## 本地一键门禁

```bash
bash scripts/verify.sh
```

脚本会从当前 Git 树（首次提交前则从排除构建/编辑器文件的目录）建立隔离副本，然后依次执行：

1. 凭据、私钥、私网地址、本机路径、固定密码、构建产物和归档扫描；
2. 必需文件、生成物排除、BOM、SVG、声明与安全文案检查；
3. 无硬件源码契约测试；
4. ESP32-S3 PlatformIO / ESP-IDF 干净构建；
5. 再次执行公开结构与秘密门禁。

它不会修改原工程或历史 ZIP，不会烧录板卡、创建真实 AP、连接浏览器、发送网络请求、点燃火焰、释放气体、运行水泵/风扇/舵机，或证明任何安全相关行为。

## 已验证的公开候选构建

2026-07-17，本地隔离门禁已在 PlatformIO Core `6.1.19`、`espressif32@6.13.0`、ESP-IDF `5.5.3`、`esp32-s3-devkitc-1` 环境通过。固件输出为：

```text
RAM:   38208 / 327680 bytes (11.7%)
Flash: 862753 / 1048576 bytes (82.3%)
```

Flash 尚余约 18%，应视为后续功能增加、页面体积和板型/分区变更时的容量边界。此本地结果用于候选审查；最终可对外固定引用的构建证据必须是默认分支 exact HEAD 对应的 GitHub Actions 成功运行。

构建结果只说明固定工具链能编译该源码。它不等同于烧录成功、设备启动、可用硬件、传感器精度、执行器反馈、网络连接、安全报警、自动处置或长期稳定性。

## 当前提交真机复测清单

未来若要更新 Hardware Lab 状态，必须将每项结果绑定到日期、完整 Git commit、具体板型、Flash、模块型号、供电方式和通过/失败/未测。建议最少覆盖：

1. ESP32-S3 开发板型号、Flash、USB、供电、当前 commit 烧录和启动日志；
2. MQ-2/MQ-5 的完整模块型号、预热时间、供电、电平、环境基线、受控变化和原始 ADC；不将结果写成 ppm 或合规报警；
3. 火焰模块型号、供电、低电平触发、环境光干扰、失败路径；不把模块输入写成真实火警；
4. HC-SR04 精确型号、ECHO 电平转换、测距范围、超时和雷达舵机耦合；
5. OLED 地址、供电、初始化、刷新和错误路径；
6. LED/蜂鸣器的实际电平、功耗、驱动和可见/可听响应；
7. 两路舵机、风扇和水泵的电源、共地、方向、限流、浪涌与停止状态；首次测试必须在断开危险负载或受控低压条件下进行；
8. 固件启动自检时每个执行器的实际动作、意外动作和复位情况；
9. 若启用本地网络，确认本地热点、浏览器、`/api/status`、SSE 断开重连和无配置时服务不启动；不要公开 SSID、密码、私网 IP 或 MAC；
10. 30–60 分钟受控低压运行，记录重启、内存、温升、传感器漂移、电源异常和网络稳定性；
11. 实物照片、视频、日志和截图去除 SSID、密码、私网、MAC、位置、设备标签和 EXIF/GPS。

即使全部清单完成，也需要独立的工程、法规和安全评估；不能将本科教学原型升级为消防、燃气、生命安全或生产设备。
